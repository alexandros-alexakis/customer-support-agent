# Decision Table - Tier 1 Issue Handling

## Overview

This table converts issue handling into structured operational logic. Each row defines what to do, what not to do, when to escalate, and where to route. This is not descriptive guidance - it is the decision layer agents and the system prompt enforce.

---

## Decision Table

| Issue Type | Typical Player Intent | Required Information | Tier 1 Allowed Actions | Prohibited Actions | Escalate? | Escalation Trigger | Escalation Target | Priority | Internal Notes Required |
|---|---|---|---|---|---|---|---|---|---|
| Payment issue - charged, no item | Delivery of paid item | Player ID, Transaction ID, Purchase date, Platform | Restart app check, Verify platform delivery delay | Promise delivery, Confirm restoration before review | Yes - if restart fails | Item still missing after restart | Billing specialist | High | Transaction ID, platform, steps attempted |
| Missing purchase - no receipt | Item or currency delivery | Player ID, Description of purchase, Approximate date | Guide player to find receipt, Check platform purchase history | Accept claim without any transaction evidence | Yes - after receipt search | Cannot locate any transaction reference | Billing specialist | Standard | Note: no receipt provided, steps to locate attempted |
| Refund request | Money returned | Player ID, Transaction ID, Platform, Whether item was used | Acknowledge request, Collect details | Approve or deny refund, State refund likelihood | Always | Refunds not handled at Tier 1 | Billing specialist | Standard | Platform type affects eligibility - note it |
| Account access - cannot login | Regain account access | Player ID, Login method, Error message | Guide through password reset, Clear cache retry | Ask for password | Yes - if reset fails | Reset email not received or method not applicable | Account team | Standard | Log login method - critical for account team |
| Suspected compromised account | Secure account, recover losses | Player ID, Login method, Date noticed, What changed | Advise immediate password change on secure device, Advise revoke third-party access | Investigate access logs, Confirm or deny breach | Always | Security issue - always specialist | Security team | High | Collect timeline of changes player did not make |
| Bug report - isolated | Fix or explain bug | Player ID, Device model, OS version, Reproduction steps | Collect details, Attempt basic troubleshooting | Confirm bug is real without verification | Yes - after collection | Standard bug escalation after collection | Technical team | Standard | Include full device details and reproduction steps |
| Possible widespread incident | Confirmation, fix ETA | Player ID, Device details, Exact behaviour | Collect report, Check if others reporting same | Speculate on cause, Promise fix timeline | Yes - immediately | Multiple similar reports in short window | Technical team + incident flag | High | Flag as potential incident. Do not wait for confirmation. |
| Technical issue - crashes or connectivity | App working again | Player ID, Device model, OS version, App version | Cache clear, Restart, Reinstall guidance | Diagnose backend issues | Yes - if troubleshooting fails | Issue persists after reinstall | Technical team | Standard | Include all troubleshooting steps already attempted |
| Harassment or abuse report | Action taken against offending player | Reporter Player ID, Reported player ID or name, Date, Description | Collect report details | Confirm what action will be taken, Share any information about reported player | Always | Trust and Safety scope | Trust and Safety | High | Note: do not share outcome of investigation with reporter |
| Ban or enforcement appeal | Account restored or explained | Player ID, Date ban noticed, Prior warnings received | Acknowledge and collect details | State whether ban was correct, Interpret TOS, Promise outcome | Always | Ban decisions not at Tier 1 | Trust and Safety | Standard | Do not characterise whether ban was fair |
| TOS question or dispute | Understand or challenge rules | Player ID if relevant | Direct to TOS document link only | Interpret, summarise, or paraphrase TOS | Yes - if player disputes enforcement | TOS interpretation not at Tier 1 | Trust and Safety | Standard | Quote directly or escalate - no paraphrasing |
| GDPR / privacy request | Data access, deletion, or correction | Player ID, Account email, Request type | Acknowledge, log receipt date, Collect details | Fulfill request, State what data is held | Always | Legal obligation - always specialist | Legal / Compliance | High | Log receipt date immediately - 30-day clock starts now |
| Game mechanic question | Understand how feature works | Player ID if issue suspected | Answer from knowledge base only | Speculate on undocumented mechanics, Comment on future changes | Yes - if not in KB | Mechanic not documented in knowledge base | Game specialist | Low | If player suspects mechanic is broken, collect details and escalate to technical |
| Duplicate or repeat contact | Resolution of previously unresolved issue | Player ID, Previous ticket reference if available | Acknowledge repeat, Review history, Escalate with full context | Treat as new ticket without checking history | Always | Third or more contact on same issue | Senior agent | High | Attach all prior contact history. This is a process failure, not just a ticket. |
| Unclear or mixed issue type | Unknown | Player ID, Clarifying question response | Ask one clarifying question | Guess at intent, Attempt resolution without classification | Yes - if still unclear after one clarifying question | Classification not possible after one attempt | Senior agent | Standard | Note what information was collected and what remains unclear |
